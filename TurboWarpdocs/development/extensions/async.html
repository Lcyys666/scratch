处理非即时操作

有时您会希望您的扩展等待"异步"操作完成后再继续执行（而不是提前继续执行并可能导致问题）。一个常见的例子是发送请求：无论您的网络有多快，任何网络请求都不可能是即时的。

现代 JavaScript 有一个很酷的工具来处理这些情况：Promise。通过这个工具，您的积木块可以告知 Scratch 它正在等待异步操作完成，以及最终的值。

async.js - Try this extension
class AsyncExtension {
  getInfo() {
    return {
      id: 'asyncexample',
      name: 'Async Blocks',
      blocks: [
        {
          opcode: 'wait',
          text: 'wait [TIME] seconds',
          blockType: Scratch.BlockType.COMMAND,
          arguments: {
            TIME: {
              type: Scratch.ArgumentType.NUMBER,
              defaultValue: 1
            }
          }
        },
        {
          opcode: 'fetch',
          text: 'fetch [URL]',
          blockType: Scratch.BlockType.REPORTER,
          arguments: {
            URL: {
              type: Scratch.ArgumentType.STRING,
              defaultValue: 'https://extensions.turbowarp.org/hello.txt'
            }
          }
        }
      ]
    };
  }

  wait (args) {
    return new Promise((resolve, reject) => {
      const timeInMilliseconds = args.TIME * 1000;
      setTimeout(() => {
        resolve();
      }, timeInMilliseconds);
    });
  }

  fetch (args) {
    return fetch(args.URL)
      .then((response) => {
        return response.text();
      })
      .catch((error) => {
        console.error(error);
        return 'Uh oh! Something went wrong.';
      });
  }
}
Scratch.extensions.register(new AsyncExtension());

请注意，异步积木块在 getInfo() 中的定义与其他积木块相同。唯一的区别是函数的返回值。

这个示例中的第一个积木块大致相当于 Scratch 的内置等待积木块。请注意，这次我们必须手动构造一个 Promise，因为 setTimeout 不使用 Promise。当 Scratch 执行到这个积木块时，它不会继续执行脚本，直到这个 promise 解决。在此期间，Scratch 可能会继续执行其他脚本。等待是一个命令积木块（没有返回值），所以它不应该解决任何值（即使有也会被忽略）。

第二个积木块尝试获取一个 URL，类似于 https://extensions.turbowarp.org/ 上的 "Fetch" 扩展。fetch() 已经返回一个 promise，所以这次我们只需要在它上面链式调用，而不是创建一个新的 promise。

返回值的限制仍然适用：您仍然需要确保返回一个字符串、数字或布尔值，所以我们在 then() 中调用 text() 将 fetch() 的 Response 对象转换为字符串。

这也演示了错误处理。每当您的 promise 可能被拒绝、失败或出错时，您都应该添加一个 catch()。具体如何处理这种情况将取决于您的用例，但通常您会希望将错误记录到控制台并返回一个表示错误的字符串。

练习​

  1. 创建一个积木块，每次调用时等待 100ms。例如，第一次运行时等待 100ms，然后是 200ms，然后是 300ms，然后是 400ms，以此类推。

下一步​

您可能已经意识到，即使您的积木块不返回 Promise，它们实际上也不是立即运行的，而且您的扩展无法访问许多 API。要确定原因，我们需要讨论"沙盒"到底是什么，以及它对您的扩展意味着什么。
